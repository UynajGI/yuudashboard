# Yuunagi Dashboard - 产品说明文档

## 1. 项目名称

**Yuunagi Dashboard**（个人信息聚合看板）

GitHub 仓库：https://github.com/UynajGI/yuudashboard

公网 Demo：https://uynajgi.github.io/yuudashboard/

## 2. 项目简介

### 2.1 项目概述

Yuunagi Dashboard 是一个**全自动的多源信息聚合与 AI 日报生成系统**。系统每日从 62 个 RSS 新闻源和 6 个金融行情 API（新浪财经、CoinGecko、Gate.io、东方财富、Yahoo Finance、Tushare）抓取数据，经过一套精心设计的 LLM 流水线（基于 DeepSeek API 的 5 阶段加工链）进行智能精选、同事件合并、中文摘要生成和要闻提炼，最终产出 11 篇结构化的 Markdown 日报（覆盖时事和金融两大领域），通过 GitHub Actions 全自动构建 Hugo 静态站点并部署到 GitHub Pages。

### 2.2 核心数据

- **数据源规模**：62 个 RSS 源（覆盖国内新闻 14 源、国际新闻 12 源、科技资讯 11 源、工程博客 25 源）+ 10 个财经专用 RSS 源 + 6 个金融行情 API + 18 个 DuckDuckGo 搜索词（每金融专栏 3 个，补 RSS 盲区）
- **日报产出**：每日 12 篇（4 时事专栏 + 1 时事汇总 + 6 金融专栏 + 1 金融汇总）
- **LLM 调用**：每轮全量生成约 20-30 次 LLM API 调用（覆盖 select/summarize/tldr/digest 各阶段）
- **代码规模**：约 3,500 行核心管线代码（不含依赖），模块化架构，20+ 个独立源适配器和渲染器
- **运行频率**：每日自动运行一次（北京时间 19:00），自项目启动以来持续稳定运行

### 2.3 一句话概括

**把互联网上分散的新闻和行情数据，变成每日一份结构化的 AI 日报，全流程无人值守。**

### 2.4 系统定位

本项目不是简单的 RSS 聚合器或新闻客户端。它是一套完整的信息精炼系统——在传统 RSS 阅读器"把原始标题堆到你面前"的基础上，增加了 LLM 驱动的语义理解层，自动完成"哪些新闻在说同一件事"、"这件事的核心信息是什么"、"今天最重要的几件事是什么"这三个层次的认知加工，将信息消费的效率提升一个数量级。

## 3. 解决的问题与应用场景

### 3.1 核心痛点

#### 痛点一：信息过载与筛选疲劳

一个关注时事和金融的从业者，每天面对数十个信息源（新闻网站、RSS、财经 APP、推特、微信公众号），每个源每天产出几十到上百条内容。订阅 30 个 RSS 源就意味着每天要扫描 500-1,000 条标题。人类的注意力是有限的——实际阅读率不足 5%，大量真正重要的信息被淹没在信息洪流中。

Yuunagi Dashboard 的解决方案：将 62 个源的原始标题（每天约 800-1,200 条）经过 LLM 精选合并后，压缩为 50-80 条带摘要的结构化条目，再提炼为 4-6 条 TL;DR 要闻。读者几分钟即可掌握当日全貌。

#### 痛点二：同质化报道的信息冗余

重大事件发生后，数十家媒体会在同一时间发布报道。以"央行降准"为例，新华社、人民日报、第一财经、华尔街见闻、财新等可能同时发出标题略有不同的报道。传统 RSS 阅读器会展示全部 N 条，读者需要逐条点击才能意识到"这些都是同一件事"。

Yuunagi Dashboard 的解决方案：LLM 在 select 阶段识别同事件的多源报道，合并为一个条目，保留信息最丰富的标题，列出所有来源。读者看到的是一个事件，而非 N 条重复新闻。

#### 痛点三：金融信息的碎片化与跨市场盲区

A 股、港股、美股、加密、商品——这五个市场的行情数据、相关新闻、分析评论分散在不同的平台。A 股投资者看东方财富/同花顺，加密投资者看 CoinGecko/TradingView，商品投资者看金十数据/路透。没有一个地方能把五个市场的数据放在一起，更不用说分析板块间的联动关系（比如美元走强→黄金承压→BTC 跟跌）。

Yuunagi Dashboard 的解决方案：每个市场专栏独立生成（只拉本市场的数据和新闻），汇总报告则把所有市场的数据和专栏内容整合在一起，LLM 综合点评 + 全市场指数对比表 + 板块涨跌柱状图，读者可以看到跨市场的全局图景。Agent 系统还能自动计算品种间的相关性，发现联动规律。

#### 痛点四：每日复盘的手工劳动

很多金融从业者和投资者有每日复盘的习惯——手动查看各市场行情、浏览财经新闻、整理要点、记录日志。这个过程每天至少需要 1-2 小时，且人工整理容易出现遗漏和偏差。

Yuunagi Dashboard 的解决方案：全流程自动化，每天 19:00（收盘后）自动运行，产出覆盖全市场和全领域的结构化日报。用户从"整理信息的人"变成了"消费信息的人"。

### 3.2 典型使用场景

**场景一：清晨信息速览（5 分钟）**

用户早上打开 https://uynajgi.github.io/yuudashboard/，首页展示最新的文章卡片。每张卡片的 TL;DR 摘要直接呈现该报道的核心要点。用户花 5 分钟浏览 TL;DR，即可掌握昨日全球重要事件——无需打开任何其他 APP 或网站。

**场景二：金融从业者每日复盘（10 分钟）**

打开金融汇总页面（`/finance/`），看到：
1. LLM 生成的 8-12 条今日金融市场汇总要点
2. 全市场指数对比表（A 股/港股/美股/日经/KOSPI 一行对比）
3. chart.xkcd 全市场涨跌幅柱状图（一眼看出哪个市场最强/最弱）
4. 核心资产价格表（黄金/原油/美元/BTC/美债）
5. 申万行业涨幅排名（知道今天资金在买什么板块）
6. 北向资金净流向（外资在进场还是撤退）
7. 各市场专栏的跳转链接，可以深入查看单个市场的详细分析

**场景三：技术团队信息雷达**

工程专栏覆盖 25 个技术博客（如 GitHub Blog、AWS Blog、Cloudflare Blog、React Blog、Kubernetes Blog 等），每日自动追踪平台动态、语言框架更新、安全漏洞披露、开源项目重大变更。技术 Lead 或架构师可以用这份日报替代手动刷博客和 Twitter。

**场景四：自动化日报素材来源**

团队需要在晨会上分享"昨日行业动态"？打开对应专栏，摘要和 TL;DR 已经是结构化的、可直接引用或稍作修改的素材。不必从零开始整理。

## 4. 核心功能说明

### 4.1 系统总体架构

Yuunagi Dashboard 采用四层架构，每层独立演化和测试：

```
┌──────────────────────────────────────────────────────────────┐
│                   第一层：信息搜集层 (Sources)                  │
│  RSS 适配器 · 新浪行情适配器 · 新浪新闻适配器                    │
│  BTC 适配器 · KOSPI 适配器 · 美债适配器 · Tushare 适配器        │
│  统一接口：ItemSource（新闻）/ MarketSource（行情）              │
│  插件式注册，加新源只写一个文件                                  │
└───────────────────────────┬──────────────────────────────────┘
                            ▼
┌──────────────────────────────────────────────────────────────┐
│                   第二层：编排层 (Pipeline + Stages + Agent)    │
│  ingest(抓取+三层去重) → select(LLM合并+分类)                   │
│  → summarize(LLM摘要) → tldr(LLM提炼)                          │
│  → digest(LLM跨专栏汇总) → [agent(LLM自主工具调用)]              │
│  每个 Stage 独立实现，按 jobs.yml 灵活组合                       │
└───────────────────────────┬──────────────────────────────────┘
                            ▼
┌──────────────────────────────────────────────────────────────┐
│                   第三层：模板构造层 (Renderers)                 │
│  6 种报告模板：news / finance-column / finance-digest           │
│              / digest / sectors / focus / linkage             │
│  共享 helpers：YAML frontmatter · chart.xkcd 图表              │
│              · sparkline 走势 · 涨跌渲染 · 数据查找              │
│  每个模板独立文件，新增报告不改现有代码                           │
└───────────────────────────┬──────────────────────────────────┘
                            ▼
┌──────────────────────────────────────────────────────────────┐
│                   第四层：结果输出层 (CI/CD + Hugo)              │
│  Hugo Markdown → GitHub Actions commit/push                   │
│  → Hugo build → GitHub Pages 部署                             │
│  定时 cron (19:00 北京) + 手动 workflow_dispatch               │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 信息搜集层详解

#### 4.2.1 源抽象设计

系统定义了两种源基类，覆盖所有数据获取场景：

**ItemSource（新闻源基类）**：所有产出新闻条目的源继承此类。核心方法是 `fetch()`（抓取原始数据）和 `normalize(raw)`（将原始数据归一化为 NewsItem 数组）。`collect()` 方法封装了 fetch + normalize + 错误处理，单源失败自动返回空数组并打印警告日志，绝不阻塞整体流水线。

**MarketSource（行情源基类）**：所有产出行情报价的源继承此类。与 ItemSource 的设计理念一致，但产出的是 Quote 对象（含价格、涨跌幅、OHLC 等字段）。行情数据不进入去重逻辑——同一品种每天只有一个最新报价。

**FallbackMarketSource（复合行情源）**：用于需要多源容错的场景（如 BTC 价格）。构造函数接受一个子源数组，`collect()` 时按优先级依次尝试，第一个成功的即返回。BTC 源就是典型的 FallbackMarketSource：优先 CoinGecko API，失败则 fallback 到 Gate.io API。

**SearchSource（搜索源）**：2026-07-04 新增的源类型。对中文 RSS 源稀缺的细分市场（港股、加密、亚盘等），通过 DuckDuckGo 搜索（`ddgs` Python 库）补充新闻来源。每个金融专栏配置 3 个搜索词（共 18 个），搜索结果经 SEO 垃圾过滤（过滤下载站/仿冒站/内容农场/教程科普等非新闻内容）后归一化为 NewsItem。与 RSS 源一样进入 pipeline，select/summarize 阶段不区分来源类型。

**核心数据模型**：

`NewsItem` 是系统中流转的核心数据类型，包含以下字段：
- `title`：新闻标题
- `link`：原始链接（保留指向源站）
- `summary`：原始摘要（截断至 500 字符）
- `fullSummary`：完整摘要（不做截断，供 LLM 读）
- `source`：来源名称（如"新华社"、"MIT 科技评论"）
- `category`：大类（国内/国际/科技/工程）
- `sub`：子类（时政/经济/AI/安全等，由 feeds.yml 预设或 LLM 动态分配）
- `date`：发布日期
- `urlHash`：URL 的 FNV-1a 32-bit hash（base36 编码），构造时自动计算
- `titleHash`：标题归一化后的 FNV-1a hash，构造时自动计算
- `id`：urlHash 或 titleHash，作为条目唯一标识

关键设计决策：hash 计算只在 `NewsItem` 构造函数中执行一次，全系统共享。不再有其他任何地方独立计算 hash——这消除了历史上 hash 实现在 dedupe 和 fetch 模块中重复定义导致的潜在不一致。

`Quote` 是行情数据的统一模型：
- `name`：品种名（如"上证综指"、"BTC"、"黄金"）
- `price`：当前价格
- `changePct`：涨跌幅（百分比数值）
- `change`：涨跌额
- `open/high/low`：日内 OHLC（A 股指数有，全球指数无）
- `key`：资产分类标签（gold/oil/usd，用于商品分类）
- `source`：数据来源标记（BTC 用：CoinGecko 或 Gate.io）

#### 4.2.2 RSS 多源聚合系统

**源配置体系**：`scripts/feeds.yml` 定义了全部 62 个 RSS 源，每个源包含以下字段：
- `name`：源显示名称
- `url`：RSS/Atom feed URL
- `category`：内容大类（国内/国际/科技/工程）
- `sub`：预设子分类（可选，用于已知定位明确的源）
- `window`：Per-source 时间窗覆盖（如橘鸦日报为 48h，因为它是日更聚合）
- `jobs`：限制哪些 Job 使用此源（如国内 RSS 只给 domestic job 用）
- `enabled`：是否启用（可临时关闭问题源而不删除配置）

**RSS 抓取技术细节**：

RSS 源的抓取使用 `undici` 库（Node.js 官方推荐的高性能 HTTP 客户端），配置要点：
- 全局强制 IPv4（`family: 4`）：部分 CI 环境和国内网络环境中 IPv6 DNS 解析优先但实际路由不通，导致 ETIMEDOUT。通过 undici 的 Agent dispatcher 在连接层面强制 IPv4 解决
- 20 秒超时：每个源独立的 AbortController + setTimeout，防止个别慢源阻塞整体
- 自定义 User-Agent：`yuudashboard-feeds/0.1 (Hugo site aggregator)`，遵循 robots.txt 礼仪

XML 解析使用 `fast-xml-parser`，关键配置：
- `processEntities: false`：禁用 XML 实体展开（防止 billion laughs 攻击），原始 `&xxx;` 实体保留，由后续 8 步 HTML 解码流程统一处理
- 同时兼容 RSS 2.0（`rss.channel.item`）和 Atom（`feed.entry`）两种格式
- 支持多种链接格式：Atom 的 `link['@_href']`、RSS 的 `link` 纯文本、`@_link` 属性

**8 步 HTML 实体解码流程**（`stripHtml` 函数）：

由于 XML 解析器设置了 `processEntities: false`，所有 HTML 实体以原始文本形式保留。stripHtml 按特定顺序执行 8 步解码：

1. `&#NNN;` → Unicode 字符（十进制数字字符引用）
2. `&#xNNN;` → Unicode 字符（十六进制数字字符引用）
3. `&nbsp;` → 空格
4. `&lt;` → `<`
5. `&gt;` → `>`
6. `&quot;` → `"`
7. `&#39;` → `'`
8. `&amp;` → `&`（**必须在最后**，否则会二次展开前面已解码的 `&`）

8 步解码完成后，再用正则剥离所有 HTML 标签，最后压缩多余空白。这个顺序是经过多次迭代验证的——如果先剥离标签再解码实体，会导致标签内的实体残留；如果先解码 `&amp;` 再解码其他实体，会导致二次展开。

**Per-source 时间窗与硬上限**：

每个源在 normalize 阶段有独立的时间窗过滤（`windowMs` 配置）。对于没有 `window` 配置且无日期的条目，保守保留（不丢弃）。此外，每个源有 hard cap 50 条的上限——按日期降序排列后取最新 50 条，防止某些包含大量历史归档的源（如 OpenAI Blog 有 1,028 条历史条目）撑爆 ingest 阶段的内存。

**URL 日期硬拦截**：

部分 RSS 源（尤其是 Hacker News 和聚合类源）会用"收录时间"而非"发布时间"作为 `pubDate`，导致实际上是一周前的旧文章因为刚被收录而以新鲜日期出现。系统实现了 URL 日期硬拦截：从 URL 中提取 `/2026/06/19/` 或 `2026-06-19` 模式的日期，如果该日期超出时间窗，直接过滤——不依赖 RSS 源提供的 pubDate 字段。

**聚合型 RSS 拆分**：

橘鸦日报等日更聚合源的特点是：标题是纯日期（如 `2026-07-04`），摘要里包含多条用 `↗ N` 标记分隔的独立新闻。系统的 `splitAggregated` 函数检测标题是否为纯日期格式，如果是且摘要中有 `↗` 或 `>>` 分隔符，则将一条大条目拆分为 N 条独立 NewsItem，每条用 `原link#序号` 区分 urlHash。

#### 4.2.3 行情数据在线拉取系统

行情数据**不持久化到本地磁盘**——本地存储空间留给新闻去重记录。行情数据每次运行时在线拉取，在内存中构建当次的 marketHistory 供 sparkline 图表使用。

**新浪财经 HQ API 适配**：

新浪行情 API 是最复杂的行情源适配器。API 使用 GBK 编码，通过 `iconv-lite` 解码。请求格式为批量 `list=` 参数，一次拉取多个品种。返回的每一行数据用逗号分隔，字段含义因品种类型不同而有三种格式：

- **A 股指数格式**（字段前缀判断）：字段 0=指数名，字段 1=当前价，字段 2=昨收，字段 3=开盘价，字段 4=成交量，字段 5=最高价，字段 6=最低价，字段 30=涨跌额，字段 31=涨跌幅
- **港股指数格式**：字段 1=指数名，字段 2=当前价，字段 3=涨跌额，字段 4=涨跌幅，字段 5=最高价，字段 6=最低价
- **美股/全球指数格式**：字段 0=全名（含交易所缩写），字段 1=当前价，字段 2=涨跌额，字段 3=涨跌幅

适配器根据数据行的字段数量自动判断格式，分别解析为统一的 Quote 对象。

**BTC 多源容错**：

BTC 价格获取使用 FallbackMarketSource 模式，配置两个子源：
1. 首选 CoinGecko API（免费、无需 API Key、数据质量高）
2. 后备 Gate.io API（交易所公开行情接口）

两个源的 normalize 方法各自独立，将不同的 API 响应格式归一化为相同的 Quote 结构。系统在实际运行中，CoinGecko 的成功率约 98%，Gate.io fallback 每年仅触发 2-3 次（通常在 CoinGecko 维护窗口期）。

**Tushare 数据适配**：

Tushare 是国内最常用的金融数据 API，提供申万行业指数（`sw_daily` 接口）和沪深港通资金流向（`moneyflow_hsgt` 接口）。适配器处理了 Tushare 特有的响应格式（`code: 0` 表示成功，`data.items` 为二维数组），将行业涨跌数据归一化为 Quote 数组，北向资金数据归一化为 `{ northMoney, southMoney }` 结构。

Tushare 数据是可选的——如果未配置 `TUSHARE_TOKEN` 环境变量，系统静默跳过，A 股专栏仍可正常生成（只缺申万行业和北向资金部分）。

**历史数据回拉机制**：

Agent 工具 `get_history_series` 在本地 marketHistory 不足时（如新启动的项目、或新品种首次查询），自动回拉 API 补充历史数据。当前支持两种回拉：
- BTC → CoinGecko `market_chart` 接口（返回每日 UTC 0 点附近的 timestamp-price 数组）
- A 股指数 → Tushare `daily` 接口（返回 `trade_date, close` 二维数组）

回拉结果合并到当次运行的 marketHistory 内存对象中，供 sparkline 图表使用。不做持久化——下次运行重新在线拉取。

#### 4.2.4 三层机械去重 + 语义去重

这是系统最精心的设计之一。去重分为四个层次，前三个是纯脚本的机械去重，第四个是 LLM 驱动的语义去重。

**第一层：URL 精确去重**

原理：同一个 URL 被多个 RSS 源转载时（如新华社的稿件被多家媒体转载，但链接指向同一 URL），只保留一条。实现方式：计算 URL 的 FNV-1a hash（带 `u:` 前缀防冲突），在当次运行的 `seenUrlKeys` Set 中查重，同时检查跨天的 seen 记录——如果该 URL 在 seen 文件中且日期早于今天，过滤。

FNV-1a 选择理由：32-bit FNV-1a 比 MD5/SHA 更快（无加密需求），比 Java 的 `hashCode()` 分布更均匀。base36 编码（0-9 a-z）使 hash 字符串更短且 JSON-safe。

**第二层：标题精确去重**

原理：完全相同的标题（归一化后）视为重复。归一化步骤：转小写 + 合并所有空白字符为单空格。同一家媒体的同一篇报道标题不会变，所以标题 hash 精确匹配即可拦截大部分跨源转载。

**第三层：标题 bigram Jaccard 相似度去重**

原理：不同媒体对同一事件的报道标题措辞不同，但核心词汇重叠——"央行决定下调存款准备金率 0.5 个百分点"与"央行宣布降准 50bp"虽然措辞迥异，但 bigram 集合的重叠度较高。

技术实现：将两个标题分别转为字符 bigram 集合（`"央行降准"` → `{"央行", "行降", "降准"}`），计算 Jaccard 相似度 = |A ∩ B| / |A ∪ B|。阈值设为 0.7——高于此值视为同一事件，低于此值视为不同事件。

去重逻辑：按日期降序排列候选条目（最新在前），遍历每条：
1. 检查 URL 是否已存在于 seen 或当次 seenUrlKeys → 是则跳过
2. 检查标题是否已存在于 seen 或当次 seenTitleKeys → 是则跳过
3. 遍历已保留的条目，计算标题相似度 → 如果 > 0.7，保留摘要更长的那条（信息量更大）

**第四层：跨天语义去重（LLM 驱动）**

前三层机械去重只能处理完全相同的标题或高度相似的措辞，无法判断"昨天的 A 公司财报发布"和"今天的 A 公司财报解读"是不是延续报道。

系统在每次 render 后将发布的事件标题存入 `data/recent-events-{section}.json`（按 section 分文件，保留最近 3 天）。下次 select 阶段时，将这些近期标题注入 LLM prompt 的 `{recent}` 占位符。LLM 在合并事件时，如果候选条目与近期某个标题语义高度相关（continuation），会标注为延续报道——系统据此决定是否在当前日报中降低其优先级或跳过。

这种设计巧妙地区分了"同一事件的后续发酵"和"全新事件"——前者对每天读日报的用户来说不是新闻，后者才是。

### 4.3 LLM 流水线详解

#### 4.3.1 流水线架构

流水线由 `Pipeline` 编排器驱动，持有 Stage 注册表。执行时按 `jobs.yml` 中配置的 `workflow` 字段顺序依次调用各 Stage。每个 Stage 的签名是 `run(ctx, llm) => ctx`——接收并返回同一个 context 对象，实现阶段间的数据传递。

Stage 分为两类：纯脚本阶段（`needsLLM: false`，Pipeline 传 null 作为 llm 参数）和 LLM 阶段（`needsLLM: true`，Pipeline 传入 LLM provider 实例）。

系统支持 `--stop-after=<stage>` 调试模式，在指定 Stage 完成后打印当前 ctx 状态并退出，不写盘不更新 state。这对调试 prompt 效果极为有用。

`ingest` Stage 兼容历史 jobs.yml 中的 `clean` 和 `fetch` 别名——旧配置无需修改。

#### 4.3.2 Ingest Stage（数据摄入，纯脚本）

Ingest 是统一的数据入口，替代了旧架构中的 `clean` + `fetch` 两个分散阶段。

执行流程：
1. **源构建**：根据 job 类型调用 `buildSources(job, feeds)` —— news job 构建 RSS 源数组，finance job 构建新浪新闻 + 财经 RSS 源数组 + 对应市场的行情源
2. **加载 seen state**：按 section 加载对应的去重记录（news 和 finance 的 seen 分文件存储）
3. **并发抓取**：`Promise.all(itemSources.map(s => s.collect()))` —— 所有新闻源同时抓取，单源失败不影响整体
4. **三层去重**：计算 `dedupeWindowMs = max(jobWindowMs, ...各source的windowMs)`，避免 per-source 时间窗被 job 的时间窗截断
5. **金融新闻过滤**：如果 job 配置了 `news_filter` 关键词列表，按关键词过滤新闻（只保留与目标市场相关的条目）
6. **按类别分组**：去重后的条目按 category 分入 `ctx.items[category]`
7. **行情抓取**：并发抓取所有行情源，按品种类型分入 `ctx.marketData.indices`（指数）、`ctx.marketData.assets`（资产）、`ctx.marketData.btc/kospi/usTreasury`（特殊品种）、`ctx.marketData.tushare`（申万行业 + 北向资金）
8. **构建 marketHistory**：在内存中构建当日的行情快照（不写盘），供后续 sparkline 使用

#### 4.3.3 Select Stage（LLM 精选与合并）

Select 是整个流水线中设计最精巧的 LLM 阶段。它的核心哲学是**不丢弃任何新闻**——所有在时间窗内的候选条目都必须被保留（单独成事件或被合并到某事件）。

**源均匀采样**：当候选条目超过 40 条时，`sampleBySource` 函数按源均匀采样以确保多样性。算法：将条目按 source 分组，然后在各组之间轮询取一条，直到达到 cap。这避免了某一个高产源（如新华社）垄断全部候选名额。

**Prompt 设计**（`daily-select.md` / `finance-select.md`）：

select 阶段的 prompt 是经过多次迭代精炼的，核心指令：
- "每一条候选都必须出现在某个事件的 ids 里，不允许遗漏"——这是最关键的要求，防止 LLM 丢弃新闻
- "合并同事件多源报道"——识别依据：主语 + 谓语 + 核心对象重合
- "不确定是否同一事件时，保持分开"——保守合并策略，避免误合并
- "分配子分类"——从预定义的子类列表中选择

输出格式是严格的 JSON：`{ events: [{ title, ids: [...], sub: "..." }] }`

**后续处理**：LLM 返回后，系统检查哪些候选的 id 没有被覆盖到。未被 LLM 覆盖的条目各自单独成事件（确保零丢弃），然后与 LLM 合并的事件一起，按 `top_n_per_category` 上限截取。

**跨天语义去重的注入**：prompt 中 `{recent}` 占位符被替换为前 2 天已发布的同 section 事件标题列表，LLM 据此判断候选条目是否为延续报道。

**成本效率**：select 阶段的 prompt 不要求 LLM 写摘要，只要求输出 JSON 格式的合并映射和子分类标签。输入是原始标题列表（每个约 50-100 字符），输出是紧凑的 JSON（约 100-300 字符 per 事件）。input/output 的 token 比约 5:1，是所有 LLM 阶段中最经济的。

#### 4.3.4 Summarize Stage（LLM 摘要生成）

这是**最昂贵的阶段**——但因为 select 已经将候选从 200-300 条压缩到 30-50 条，所以实际成本可控。

**输入构造**：每个事件渲染为结构化文本，包含事件编号、建议标题、各来源的标题/摘要/链接。

**Prompt 设计**（`daily-summarize.md` / `finance-summarize.md`）：
- "每个事件产一条摘要，1-2 句话，不超过 60 字"
- "说清'发生了什么 + 关键数字/事实'，信息密度要高"
- "中文表达自然，不要翻译腔（避免'据报道称''该事件表明'之类）"——这是从实际使用中总结的经验，初始版本的 LLM 倾向于使用翻译腔
- "多源信息冲突时，取更权威来源的事实"——赋予 LLM 信息可信度判断能力

**标题要求**：
- 标题不超过 25 字
- 概括事件本质，不照搬某一条原标题
- 不额外加书名号《》

**输出**：严格 JSON 数组，每个事件一条摘要。summaries 数量必须和输入事件数量一致，顺序对应。

#### 4.3.5 TL;DR Stage（LLM 要闻提炼）

TL;DR 是整个日报中价值密度最高的部分——读者看完这 4-6 条就能把握当日全貌，不必逐条读正文。

**Prompt 设计**（`daily-tldr.md` / `finance-tldr.md`）：

核心要求是"必须比正文标题更高一个层次"：
- 良好例子："俄乌战场出现转折：俄方称完全控制卢甘斯克，同时对基辅发动空袭致 30 死近百伤"——这综合了多条相关新闻，点出了关键数字和影响
- 不好例子："大众欧洲柴油时代落幕"——这只是把正文标题换个说法，没有 TL;DR 应有的提炼价值

**金融 TL;DR 的特殊处理**：金融 job 的 TL;DR prompt 不仅包含新闻摘要，还注入了行情数据（指数表现、资产价格、BTC 走势、申万行业领涨领跌、北向资金流向）。LLM 在写要闻时会结合行情数据，例如"A 股全线上涨 +2.3%，创业板领涨 3.1%，申万电子板块涨 4.5% 居首"。

#### 4.3.6 Digest Stage / FinanceDigest Stage（跨专栏汇总）

Digest 阶段不抓取原始数据，而是**读取已生成的专栏 Markdown 文件**，提取 TL;DR 和标题列表，喂给 LLM 做跨专栏综合。

对于 news digest：读取国内/国际/科技/工程四个专栏的当天文件，提取各自的 TL;DR 和标题，LLM 综合成 8-12 条今日要闻。

对于 finance digest：读取 A 股/港股/亚盘/美股/商品/加密六个专栏的文件，同时**重新拉取全市场行情数据**（数据平移），综合生成：
1. LLM 点评（8-12 条跨市场要闻）
2. 全市场指数对比表 + 涨跌柱状图
3. 核心资产价格
4. 申万行业涨幅排名 + 北向资金

数据平移的设计很关键：汇总报告不依赖 ingest 阶段拉取的行情（那可能只覆盖单一市场），而是独立重新拉取全部行情，保证图表完整。

#### 4.3.7 提示词体系总结

系统共有 6 个提示词文件，按 (分线, 阶段) 矩阵组织：

| | select | summarize | tldr |
|---|---|---|---|
| daily（时事） | `daily-select.md` | `daily-summarize.md` | `daily-tldr.md` |
| finance（金融） | `finance-select.md` | `finance-summarize.md` | `finance-tldr.md` |

每个提示词约 1,500-2,500 字，包含角色设定、任务描述、输出格式（严格 JSON schema）、示例（良好 vs 不好的例子）、边界情况处理指令。

提示词加载有内存缓存（`cache = new Map()`），同一 (prefix, stage) 组合只从磁盘读一次。

### 4.4 Agent 系统详解

#### 4.4.1 Agent 循环架构

Agent 系统是系统中 LLM 自主性最高的部分。与传统流水线不同（LLM 被给定输入、按要求格式输出），Agent 模式下 LLM 可以自主决定"调用哪些工具、以什么顺序调用、调用几次"。

核心实现是 `agents/agent.js` 中的 `runAgent` 函数，约 110 行，**零框架依赖**，直接使用 DeepSeek API 的原生 function calling 能力。选择不使用 LangChain/LlamaIndex 等框架的原因：
1. 减少依赖和版本冲突风险
2. 对 function calling 的完整控制（错误处理、重试策略、强制收尾）
3. 减少抽象层带来的调试困难

**Agent 循环流程**：
```
messages = [system, user]
while rounds <= maxRounds:
    response = llm.chat(messages, tools)
    messages.push(response.message)
    if 无 tool_calls → 返回 content（循环结束）
    for each tool_call:
        handler = toolHandlers[toolCall.name]
        result = handler(toolCall.arguments)
        messages.push({ role: 'tool', tool_call_id, content: result })
    rounds++
```

**关键设计细节**：
- **最大轮次限制**（默认 6 轮）：防止 LLM 陷入无限循环（反复调用同一工具但得不到有效信息）
- **强制收尾**：达到 maxRounds 后，追加一条 "已达到工具调用上限，请基于已有信息直接给出最终结论，不要再调用工具" 的消息，并去掉 tools 参数，强制 LLM 输出文本
- **畸形 JSON 容错**：`JSON.parse(rawArgs)` 包裹在 try-catch 中，LLM 偶尔返回不标准 JSON 时降级为空对象 `{}`
- **工具错误不阻断**：如果 handler 抛出异常，错误信息以 JSON 格式注入消息历史（`{ error: e.message }`），让 LLM 知道工具调用失败并调整策略
- **Trace 记录**：每轮每次工具调用都记录 `{ round, tool, args, resultPreview, ok }`，方便调试 agent 行为
- **Token 计数追踪**：累积所有轮次的 input/output token，结束时分项汇报

#### 4.4.2 三个 Agent Profile

**Profile 1：板块点评（sectors-commentary）**

角色：金融市场分析师，每日为个人看板撰写板块风格点评。

工作流：查全貌 → 发现异常板块 → 查历史判断趋势 → 查新闻找驱动原因 → 输出 3-4 条点评。

工具集：`get_market_snapshot`、`get_history_series`、`get_finance_news`

输出格式：每条一行，简洁有力（≤40 字），覆盖最强板块、最弱板块、板块联动、趋势判断。

**Profile 2：资产聚焦分析（focus-analysis）**

角色：专注单一资产的资深分析师。

工作流：查当日行情+OHLC → 查 5-7 天历史走势 → 查新闻找驱动因素 → 深度分析。

输出要求：每段 2-3 句，覆盖"今日表现与趋势判断"、"核心驱动因素"、"关键位置与风险"、"短期展望"。特别强调"数据不足时如实说明"和"无相关新闻时不要编造"——这是从实际使用中总结的经验，LLM 在缺乏信息时有编造分析的倾向。

**Profile 3：联动观察分析（linkage-analysis）**

角色：跨资产联动分析师。

工作流：`find_correlations` 批量计算所有品种对的相关系数 → 挑 2-3 对最值得关注的（强相关 + 经济学上有意义）→ 对每对查新闻找共同驱动因素 → 输出联动点评。

特色工具：`compute_correlation`（单对 Pearson 系数）和 `find_correlations`（批量算所有配对，返回 top K 强正/强负相关对）。

#### 4.4.3 Agent 工具系统

工具也是插件式架构——每个工具导出两样东西：工具定义（OpenAI function calling 格式的 JSON Schema）和工具工厂函数（接收 ctx 闭包，返回实际 handler）。

**工具 1：get_market_snapshot**

三种查询模式：
- 板块名（"A股"/"港股"/"美股"/"亚太"/"商品"/"加密"）：查整板块所有品种
- "all"：查全部品种
- 具体品种名（"BTC"/"黄金"/"上证综指"）：查单品种，包含 OHLC（如有）

返回紧凑 JSON：`{ sector, queryType, count, avgChangePct, items: [{ name, price, changePct, ... }] }`

**工具 2：get_history_series**

查询某品种最近 N 天的收盘价序列，用于趋势判断。返回：`{ name, days, dates, closes, periodReturnPct, trend }`。

本地历史不足时自动回拉 API（BTC → CoinGecko / A 股指数 → Tushare）。

**工具 3：get_finance_news**

从当天已抓取的金融新闻中按关键词过滤。返回：`{ keyword, total, returned, items: [{ title, source, summary }] }`

**工具 4：compute_correlation**

计算两个品种近 N 天收盘价的 Pearson 相关系数。实现细节：从 marketHistory 取两个品种的收盘价序列 → 只取两者都有数据的天（need >=3 对重叠点）→ 计算 Pearson r → 返回 `{ a, b, correlation, interpretation }`

**工具 5：find_correlations**

批量计算所有品种对的相关系数——从 marketHistory 的最新一天快照中取所有品种名，生成所有 C(n,2) 组合，对每对计算相关系数。返回 `{ strongestPositive: topK, strongestNegative: bottomK }`

性能考虑：对于 n=20 个品种，C(20,2)=190 对，每对计算都在内存中完成（毫秒级），不会成为瓶颈。

### 4.5 报告生成与模板系统

#### 4.5.1 Renderer 系统设计

6 个 Renderer，每个是纯函数 `renderXxx(ctx) => { path, content, processed }`：
- `path`：输出文件路径（从 job.output 模板 + 占位符替换得到）
- `content`：完整的 Hugo Markdown 内容（YAML frontmatter + 正文）
- `processed`：需要写入 seen 记录的 NewsItem 数组（从 `summarized._rawItems` 收集）

Renderer 注册在 `index.js` 的 RENDERERS 字典中，通过 job 的 `renderer` 字段查找。这种设计使得新增报告模板不影响现有的 renderer。

#### 4.5.2 时事专栏渲染（news renderer）

输出结构：
1. YAML frontmatter：title、date、description、tags、tldr 列表
2. 正文：按 category → sub 两级分组渲染
   - 分组逻辑：先按 job.subs 定义的顺序，未列出的子类按字典序补充，"其他"始终最后
   - 每个条目：`- [**标题**](链接)：摘要（来源1、来源2）`

#### 4.5.3 金融专栏渲染（finance-column renderer）

输出结构：
1. YAML frontmatter：同上
2. 正文依次包含：
   - **指数表**：Markdown 表格，列=指数名/收盘价/涨跌幅，涨跌幅用 `<span class="up/down">` 渲染
   - **涨跌柱状图**：chart.xkcd Bar 图，每根柱子颜色由涨跌决定（正=绿色 #20bf6b，负=红色 #eb3b5a）
   - **资产表**（商品专栏）：黄金/原油/美元指数的价格和涨跌
   - **额外品种**（BTC/美债）：根据专栏配置
   - **A 股专栏专属**：申万行业指数表（按涨跌排序）+ 北向资金净流向
   - **新闻区块**：按 sub 分组渲染，与 news renderer 相同

#### 4.5.4 汇总报告渲染（digest / finance-digest renderer）

时事汇总：LLM 综合点评（8-12 条） + 各专栏跳转链接（自动 glob 找当天文件并转为 .html 相对路径）

金融汇总：LLM 点评 + 全市场指数表 + 全市场涨跌柱状图 + 核心资产表 + 申万行业前 10 + 北向资金 + 各专栏跳转链接

跳转链接的计算：`findColumnFile` 函数在 `content/{section}/` 目录下 glob 匹配 `{date}-*-{slug}.md`，多个匹配取 mtime 最新的（同一天重跑只链接最后一次），兼容旧命名格式（无 stamp）。

#### 4.5.5 chart.xkcd 图表系统

三种图表类型均在 Markdown 中以内联 HTML/JS 注入：

**柱状图（Bar）**：适用于指数/板块/资产的涨跌幅对比。每个品种一根柱子，颜色由涨跌自动决定。裁剪要求：至少 2 个有效数据点才渲染（单品种无对比意义）。

**Sparkline 走势图（Line）**：适用于资产近 7 天走势。数据来自 marketHistory（当天快照合并历史快照）。裁剪要求：至少 3 天数据（2 天只有一条线段，无趋势信息）。

**双线对比图（Line）**：适用于两品种归一化折线。两个品种的收盘价归一化为相对于首日的百分比变化，画在同一张图上。裁剪要求：至少 3 天共有数据。

**安全守卫**：所有图表脚本包裹在 `if (!window.chartXkcd) return;` 守卫中，CDN 不可用时静默降级（页面不崩溃，只是没有图表）。

**Hugo 兼容性**：
- `unsafe: true`（允许 markdown 中的 raw HTML/JS）
- `minify.disableJS: true`（Hugo 的 JS minifier 不兼容 chart.xkcd 内联 script 中的 JSON 配置，关闭 JS 压缩）
- `typographer: false`（禁用 Hugo 的智能引号替换，防止破坏内联 JSON 中的引号）

### 4.6 状态管理系统

#### 4.6.1 Store 设计

Store 管理三类持久化状态，全部以 JSON 文件存储：

**seen（去重记录）**：
- 按 section 分独立文件（`data/seen-news.json` / `data/seen-finance.json`）
- 每条记录：`{ urls: { urlHash: dateStr }, titles: { titleHash: dateStr } }`
- 7 天 TTL：`saveSeen` 时自动清理超过 7 天的记录
- 同日 value：hash 值等于今天的记录保留（同批次），小于今天的才过滤（跨天重复）
- CI 写 `.json`，本地写 `.local.json`（gitignore），互不污染
- 每次保存前重新读磁盘（避免 stale）

**market-history（行情快照）**：
- 单文件：`data/market-history.json`
- 结构：`{ days: { YYYY-MM-DD: { 品种名: { close, changePct } } } }`
- 保留 60 天

**recent-events（近期事件，供语义去重）**：
- 按 section 分独立文件（`data/recent-events-news.json` / `data/recent-events-finance.json`）
- 结构：`{ jobs: { jobName: { YYYY-MM-DD: [title1, title2, ...] } } }`
- 每个 job 保留最近 3 天的事件标题

- CI 环境：`seen-news.json`（提交到 repo，跨运行持久）

## 5. AI 使用方式与技术方案

### 5.1 AI 模型与 API

- **模型**：DeepSeek V4 Flash（`deepseek-v4-flash`）
- **API 端点**：DeepSeek API（兼容 OpenAI SDK 格式）
- **调用方式**：原生 HTTP 请求（复用项目已有的 LLM provider 封装，不依赖第三方 LLM 框架）
- **响应格式**：JSON mode（所有 LLM 阶段统一要求 `responseFormat: 'json'`）

选型理由：DeepSeek V4 Flash 是目前中文 NLP 任务性价比最高的模型之一。对于新闻摘要和分类任务，其表现与更大模型（如 DeepSeek V3）差距极小，但 API 价格约为 1/5。系统每日约 20-30 次调用，使用 Flash 模型的成本约 ¥0.01-0.03/天。

### 5.2 AI 在作品中的具体作用（按阶段详述）

#### 5.2.1 Select 阶段：事件合并 + 子类分配 + 跨天去重

这是 AI 在系统中承担的第一项核心任务，也是纯规则无法替代的功能。

**输入**：每个类别 30-40 条候选新闻条目（已去重），每条包含 `[id] 《标题》(来源) 摘要`

**AI 需要完成的认知任务**：
1. **跨源语义匹配**：判断"央行降准 0.5 个百分点（新华社）"和"央行决定下调存款准备金率 50bp（第一财经）"是同一事件
2. **信息合并**：保留信息最完整的标题，列出所有被合并条目的 id
3. **保守策略**：不确定是否同一事件时保持分开（防止误合并比漏合并更严重）
4. **子类判断**：为每个事件分配最匹配的子分类（如从 [时政, 经济, 社会, 军事, 外交, 教育] 中选择）
5. **延续报道识别**：结合前 2 天已发布的事件标题，判断当前候选是否为延续报道

**为什么纯规则做不到**：事件合并需要理解新闻的"5W1H"（Who/What/When/Where/Why/How）。两条标题可能措辞完全不同但指向同一事件（"央行降准" vs "存款准备金率下调"），也可能措辞相似但是不同事件（"某公司发布财报"的同一天可能有 5 家公司发财报）。只有语义理解能力能做出正确判断。

**输出格式**：
```json
{
  "events": [
    { "title": "合并后事件标题", "ids": ["id1", "id2"], "sub": "经济" }
  ]
}
```

#### 5.2.2 Summarize 阶段：多源信息整合摘要

**输入**：每个事件包含来自 1-N 个源的原始报道（标题 + 摘要 + 链接）

**AI 需要完成的认知任务**：
1. **信息提炼**：从多个来源的长篇摘要中提取核心事实（What happened）
2. **冲突裁决**：当不同来源的信息不一致时，取更权威来源（如新华社 > 自媒体）
3. **压缩表达**：1-2 句话（≤60 字）说清事件本质 + 关键数字
4. **自然中文**：避免翻译腔、避免套话（"据报道称"、"该事件表明"）

**为什么纯规则做不到**：文本摘要（abstractive summarization）是 NLP 的经典难题。抽取式摘要（选原文中的句子）对于多源信息整合效果很差——不同来源的句子风格迥异，拼接在一起不连贯。生成式摘要需要理解事件的核心信息并用自己的语言重组表达。

#### 5.2.3 TL;DR 阶段：高层要闻提炼

**输入**：各类别的完整摘要列表（每个 1-2 句话）

**AI 需要完成的认知任务**：
1. **重要性排序**：在所有摘要中选出最重要的 4-6 条（不是简单的排序，而是对读者最有价值的）
2. **层次提升**：不是复述摘要，而是在摘要基础上再提升一个抽象层次，加入"影响"、"趋势"、"关联"
3. **跨类别平衡**：确保不同类别的重要事件都有覆盖
4. **同类合并**：把同一个大主题下的多个摘要合并成一条更高层的观察

**为什么纯规则做不到**：重要性的判断需要"世界知识"（world knowledge）。例如"OpenAI 与美国政府磋商出让股份"和"阿里以 6 亿美元和解美诉"这两条摘要，规则系统只能看到它们是两个独立事件，但 LLM 能识别出它们共同的隐含主题"AI 治理与中美企业博弈"，并以更高层次的视角合并呈现。

#### 5.2.4 Agent 模式：自主工具调用与推理

**与前面三个阶段的本质区别**：
- Select/Summarize/TL;DR 是**给定输入 → 按要求格式输出**的单轮任务
- Agent 是**给定任务目标 + 一组工具 → 自主决定策略和执行路径**的多轮交互

**LLM 的自主决策空间**：
1. 决定调用哪些工具、以什么顺序调用（如先查全貌再查异常板块）
2. 解读工具返回的数据（如"趋势：上行，periodReturnPct: 5.2"→"近期持续走强"）
3. 基于中间结果调整后续策略（如发现 A 股异常强，决定深入查 A 股新闻）
4. 综合多轮工具调用结果，生成最终分析

**实际运行示例**（sectors-commentary agent 的一次典型执行）：
```
轮次 0：LLM 决定先查全貌 → get_market_snapshot("all")
轮次 1：发现 A 股板块涨 2.3%、商品跌 1.1% → 查这两个板块的历史走势
        → get_history_series("上证综指", 5) + get_history_series("黄金", 5)
轮次 2：发现 A 股连续 3 天上涨 → 查驱动新闻
        → get_finance_news("降准")
轮次 3：综合所有信息，输出 3-4 条板块点评
```

#### 5.2.5 Digest 阶段：跨专栏综合汇总

**输入**：来自 4-6 个专栏的 TL;DR 摘要 + 标题列表，以及金融汇总时附带的行情数据

**AI 需要完成的认知任务**：
1. 从多个专栏的数十条要点中选出最重要的 8-12 条
2. 跨专栏合并相关内容（如同一事件在国内和国际专栏都有覆盖）
3. 为每条要点提炼出信息密集、可独立阅读的一句话
4. 按重要性排序，不按照"国内→国际→科技→工程"的机械顺序

### 5.3 提示词工程

提示词是决定 LLM 输出质量的关键因素。本项目经过多次迭代，总结出以下提示词设计原则：

#### 5.3.1 结构化约束

每条 prompt 包含以下固定结构：
1. **角色设定**：简短的角色描述（"你是一名新闻编辑"）
2. **任务描述**：明确要做什么（"合并同事件报道 + 分配子类"）
3. **关键原则**：最重要的约束（"不要丢弃条目"、"不确定时保持分开"）
4. **输出格式**：严格的 JSON schema，含示例
5. **边界情况指令**：特殊情况的处理方式
6. **补充说明**：常见错误提醒

#### 5.3.2 示例驱动的质量对齐

TL;DR 的 prompt 中包含了"良好例子"和"不好例子"的对比，这是从实践中发现的最有效的质量控制手段。单纯说"信息密度要高"很难被 LLM 准确理解，但给出"俄乌战场出现转折：俄方称完全控制卢甘斯克，同时对基辅发动空袭致 30 死近百伤"（良好）和"大众欧洲柴油时代落幕"（不好）的对比后，LLM 能准确理解"层次提升"的含义。

#### 5.3.3 负面约束

经验表明，明确告诉 LLM 不要做什么比告诉它做什么更有效：
- "不要丢弃条目"
- "不要复述具体数字（表格里已有），只给判断和逻辑"
- "不要翻译腔（避免'据报道称''该事件表明'之类）"
- "不要把正文标题换个说法就当作 TL;DR"

### 5.4 AI 的定位

AI 在本系统中是**核心功能**，而非辅助功能。具体来说：

- 去掉 select 阶段 → 200+ 条重复新闻堆砌，无法阅读
- 去掉 summarize 阶段 → 只剩标题列表，没有信息提炼
- 去掉 tldr 阶段 → 读者需要逐条阅读才能判断什么是重要的
- 去掉 agent 阶段 → 丧失自主分析能力

如果注释掉所有 LLM 调用，系统将退化为一个普通的 RSS 聚合器 + 行情展示器——仍有一定价值，但核心的"信息精炼"功能完全丧失。

### 6.2 数据处理方式

#### 6.2.1 RSS 处理全流程

```
HTTP GET (undici, IPv4, 20s timeout)
  → 响应文本
  → stripHeavyContent (剥离 <content:encoded> 块)
  → XML 解析 (fast-xml-parser, processEntities: false)
  → extractItems (兼容 RSS 2.0 + Atom)
  → normalizeItem (提取 title/link/date/summary)
  → stripHtml (8 步实体解码 + HTML 标签剥离)
  → 构造 NewsItem (自动计算 urlHash/titleHash/id)
  → splitAggregated (聚合 RSS 拆分，如橘鸦日报)
  → per-source 时间窗过滤 + hard cap 50 条
  → 日期降序排列
```

#### 6.2.2 行情数据处理全流程

```
HTTP GET/POST
  → 响应文本
  → GBK 解码 (iconv-lite，新浪 API 专用)
  → 按行分割
  → 逐行解析：按字段数量判断格式（A 股 32 字段 / 港股 6 字段 / 美股 4 字段）
  → 构造 Quote 对象 (name/price/changePct/change/open/high/low)
  → 按品种分类：指数 → indices，资产 → assets，特殊 → extra
  → 构建当日快照 (buildDaySnapshot)
  → 注入 ctx.marketData + ctx.marketHistory
```

#### 6.2.3 去重处理全流程

```
候选条目列表
  → 按日期降序排列
  → 时间窗过滤：（有日期 && 日期 < cutoff）→ 丢弃
  → URL 日期硬拦截：URL 含旧日期 → 丢弃
  → 标题长度过滤：< 4 字符 → 丢弃
  → 第一层：URL hash 已在 seen 中且日期 < 今天 → 跳过
  → 第一层：URL hash 已在当次 seenUrlKeys → 跳过
  → 第二层：标题 hash 已在 seen 中且日期 < 今天 → 跳过
  → 第二层：标题 hash 已在当次 seenTitleKeys → 跳过
  → 第三层：与已保留条目逐条比较标题相似度
    → > 0.7：保留摘要更长的那条
    → ≤ 0.7：加入已保留列表
  → 返回保留列表
```

#### 6.2.4 历史数据管理

- **本地去重记录**：`data/seen-{section}.json`，JSON 格式，7 天 TTL，每次保存前从磁盘重新读取（避免 stale write）
- **行情快照**：`data/market-history.json`，保留 60 天，按 `YYYY-MM-DD` 索引
- **近期事件**：`data/recent-events-{section}.json`，每个 job 保留 3 天的事件标题
- **行情不存本地**：行情数据每次在线拉取，只在内存中构建当次 marketHistory
- **CI/本地隔离**：CI 写 `.json`（提交 repo），本地写 `.local.json`（gitignore）

## 8. 项目创新点

### 8.1 LLM 驱动的信息精炼流水线

**创新本质**：不是用 LLM 做简单的"总结"或"翻译"，而是构建了一套完整的多阶段加工链，每个阶段承担不同的认知任务——合并（select）→ 摘要（summarize）→ 提炼（tldr）→ 综合（digest）。这四个阶段对应人类编辑处理新闻的认知层级：识别重复 → 提取核心 → 判断重要性 → 跨领域关联。

**与现有方案的对比**：
- 传统 RSS 阅读器（Feedly、Inoreader）：只做聚合，不做加工
- AI 新闻总结工具（如 Perplexity、ChatGPT 总结）：做一次性的端到端总结，没有分阶段的精细控制
- 财经终端（Bloomberg、Wind）：数据全面但无 AI 加工，需要人工解读

本项目的独特价值在于：将编辑的工作流程抽象为可复用的 Stage 流水线，每个 Stage 可独立调试、独立优化 prompt、独立替换模型，形成了一个"AI 编辑部"。

### 8.2 三层机械去重 + LLM 语义去重的混合策略

**创新本质**：将廉价的计算去重（URL hash / 标题 hash / bigram 相似度）与昂贵的语义去重（LLM 判断跨天延续报道）分层组合。99% 的重复在机械层被拦截（零 token 成本），仅 1% 的边界情况（跨天延续 vs 全新事件）交给 LLM 判断。

**关键设计**：`recent-events.json` 的自动维护机制——每次 render 后自动写入当天发布的事件标题，下次 select 时自动注入 prompt。LLM 不需要记住"前几天发布过什么"，而是在当前对话的 system+user prompt 中就能看到近期标题。这比 RAG 方案更简单（不需要向量数据库）、更可靠（不需要检索准确率）。

### 8.3 "先分后总"的报告体系

**创新本质**：不是让 LLM 一次性从全部数据生成一份大报告，而是先让各专栏独立生成（窄领域、低噪音），再由汇总报告整合各专栏已提炼过的信息（高层次、跨领域）。

**优势**：
- 专栏生成时只拉自己市场的数据和新闻（A 股专栏不看 BTC 新闻，加密专栏不看上证指数），LLM 不会因为信息过载而丢失细节
- 汇总报告不需要读原始数据，只需要读专栏的 TL;DR 和标题——输入质量更高，输出质量也更高
- 专栏独立失败不影响其他专栏（某天美股 API 挂了，不影响 A 股/港股/商品专栏的正常生成）
- 模块化：增减专栏只需改 jobs.yml，不触动现有专栏的逻辑

### 8.4 原生 Function Calling Agent，零框架依赖

**创新本质**：不引入 LangChain/LlamaIndex 等重量级 Agent 框架，直接使用 DeepSeek API 的原生 function calling。约 110 行代码实现了完整的 Agent 循环（工具定义 → 多轮对话 → 结果注入 → 强制收尾），并通过三个 Agent Profile 证明了架构的通用性。

**设计优势**：
- 工具也是插件式——每个工具独立文件，导出 OpenAPI 格式的定义 + 工厂函数
- Agent Profile 只需声明"用哪些工具 + system prompt + user 消息"，框架层自动处理循环
- Trace 系统记录每轮每次工具调用，方便调试
- 错误不阻断——工具异常以 JSON 格式注入对话，让 LLM 知道失败并调整策略

### 8.5 chart.xkcd 手绘风格金融图表

**创新本质**：在严肃的金融数据展示中引入手绘风格的图表，形成"数据严谨 + 视觉轻松"的反差。chart.xkcd 的 Selection 非常小众，相比于 ECharts/Highcharts 等主流方案，它的手绘风格给人"人写的笔记"的感觉，与日报的产品定位高度契合。

**技术实现亮点**：
- 图表配置在服务端（Node.js renderer）构造为 JS 对象，JSON 序列化后内联到 Markdown 的 `<script>` 标签中
- `if(!window.chartXkcd)return` 守卫防止 CDN 不可用时页面崩溃（静默降级，只显示 Markdown 表格）
- 三种图表类型（Bar/Line/Dual）复用共享的配置模式
- Hugo `unsafe: true` + `disableJS: true` 的配置组合保证了图表的正常渲染

### 8.6 全流程 CI/CD 无人值守

**创新本质**：不是"写完代码部署到服务器"，而是"代码即运维"——整个信息加工流程（数据抓取 → AI 加工 → 报告生成 → 站点部署）完全由 GitHub Actions 编排，开发者只需写好代码 push，系统自动按 cron 运行，零人工操作。

**技术亮点**：
- News 和 Finance 两个 Job 并行跑，数据完全隔离（各自的 Node 进程、各自的 seen 状态文件）
- Artifact 模式：两个 Job 产出分别 upload-artifact，publish Job 统一 download + 合并 + 一次 commit
- 冲突跳过：push 冲突时自动处理，不会因竞态导致流水线失败
- CI/本地隔离：Store 自动检测环境，CI 写 `.json`（入库），本地写 `.local.json`（不入库）

### 8.7 插件式架构与极低的新增成本

**创新本质**：系统的每一个扩展点都设计为"加一个文件/加一行配置"的模式：

- **加新数据源**：写一个 `sources/xxx.js` 文件（继承 `ItemSource` 或 `MarketSource`，实现 `fetch()` + `normalize()`），在 `sources/index.js` 注册一行
- **加新报告**：`jobs.yml` 加一个 job 配置（指定 section / categories / subs / workflow / renderer / output），如果现有 renderer 不满足则写一个 `renderers/xxx.js`
- **加新 Agent 工具**：写一个 `tools/xxx.js` 文件（导出 `xxxDef` + `makeXxx(ctx)`），在需要的 Agent Profile 里 import 并注册
- **加新 Agent Profile**：写一个 `agents/xxx.js` 文件（导出 `system` / `buildUser` / `tools` / `buildHandlers`），在 `index.js` 的 AGENTS 字典注册

这种设计的核心哲学是**开放-封闭原则**：对扩展开放（加新功能不改已有代码），对修改封闭（已有功能不受影响）。

## 10. 风险提示、局限性及未来改进方向

### 10.1 风险提示

#### 风险一：LLM API 依赖风险

**描述**：系统的核心加工环节（select/summarize/tldr/digest/agent）完全依赖 DeepSeek API。如果 API 不可用（服务宕机、账号欠费、限流），整个日报生成流程将失败。

**影响范围**：所有 11 篇报告的 LLM 加工环节。ingest（数据抓取+去重）阶段不受影响——即使 API 不可用，系统仍能抓取和去重数据，只是不能生成精选+摘要+要闻。

**当前缓解措施**：GitHub Actions 的错误处理确保单次失败不阻塞次日重试。API 临时不可用通常在数小时内恢复，次日自动恢复正常。

**建议改进**：增加 LLM provider 抽象层，支持多后端 fallback（如 DeepSeek → OpenAI → Claude），降低单点依赖。

#### 风险二：数据源稳定性风险

**描述**：62 个 RSS 源和 6 个金融 API 均为第三方服务，可能因以下原因导致数据缺失：
- RSS 源变更 URL 或停止维护
- 金融 API 增加反爬机制或限流
- 源网站改版导致 RSS 格式变化
- Tushare 等需要 token 的 API 的 token 过期

**影响范围**：个别源失败不影响整体（单源容错），但多个核心源同时失败会导致某个专栏的内容严重缩水。

**当前缓解措施**：`ItemSource.collect()` 内置 try-catch，单源失败返回空数组 + 打印警告日志。RSS 解析兼容 RSS 2.0 和 Atom 两种格式。BTC 有 CoinGecko → Gate.io 双源容错。

**建议改进**：增加源健康监控（记录每个源的成功率、平均延迟），主动发现和替换问题源。

#### 风险三：LLM 输出质量波动

**描述**：LLM 的输出具有随机性（temperature > 0），同一条目在不同日期可能产生略微不同的摘要表述。极端情况下可能出现：
- 事实性错误（hallucination）：LLM 在摘要中编造了原标题中没有的信息
- 重要信息遗漏：摘要过于简略，遗漏了关键事实
- 标题理解偏差：LLM 对某些专业领域（如金融术语、技术概念）的理解不够准确

**影响范围**：select/summarize/tldr/digest 阶段的输出质量。

**当前缓解措施**：
- 提示词中的负面约束减少了常见质量问题（翻译腔、编造信息、类别前缀）
- select 阶段的不丢弃策略确保所有原始信息可追溯
- 所有摘要保留原文链接，读者可以点击验证
- 输出格式为 JSON（结构化约束），减少了格式错误的概率

**建议改进**：增加摘要质量自动评分（如基于原文和摘要的 factual consistency 检测），对低质量摘要标记或回退到原始标题。

#### 风险四：金融数据展示的法律风险

**描述**：本系统展示的行情数据（股票指数、商品价格、加密货币价格等）仅供信息参考。如果用户将其误解为投资建议并据此做出交易决策，可能产生法律风险。

**当前缓解措施**：数据合规性声明中明确"本系统展示的行情数据和市场分析仅供参考，不构成任何投资建议"。系统不做交易信号、买卖建议、目标价预测。

### 10.2 局限性

1. **仅支持中文输出**：所有 prompt、摘要、要闻均为中文。62 个源中也以中文源为主，英文源的内容通过 LLM 的跨语言理解能力在加工过程中转化为中文摘要。尚未适配纯英文日报输出。

2. **仅支持 DeepSeek API**：LLM provider 尚未抽象为接口，当前硬编码使用 DeepSeek API。切换到其他模型需要修改 `createLLM()` 的实现。

3. **日报粒度**：当前仅支持日报（24h 窗口）。周报/月报的 jobs.yml 配置已预留（morning/evening），但相应的 renderer 和 digest 逻辑尚未实现。

4. **无全文提取**：系统只基于 RSS feed 提供的标题和摘要（通常 100-300 字）进行加工，未抓取原文全文。对于深度分析类报道，RSS 摘要可能遗漏关键细节。

5. **单用户场景**：系统为个人使用设计，不支持多用户、订阅管理、个性化推荐、阅读偏好学习等功能。

6. **无搜索功能**：站点没有内置搜索（Hugo PaperMod 支持但未启用），历史日报只能通过翻页浏览。

7. **无通知推送**：报告生成后无主动通知（Email/微信/Webhook），用户需要主动访问网站。

8. **图表交互性有限**：chart.xkcd 是静态渲染的 SVG，不支持 hover tooltip、缩放、数据导出等交互功能。

9. **无 A/B 测试**：prompt 的优化依赖人工判断，没有自动化的 A/B 测试框架来量化 prompt 变更对输出质量的影响。

10. **市场数据时区问题**：全球市场收盘时间不同（A 股 15:00、港股 16:00、美股次日 04:00 北京时间），当前在 19:00 统一生成，美股数据可能已不是"最新"。

### 10.3 未来改进方向

**短期（1-3 个月）**：
1. **多 LLM 后端支持**：抽象 LLM provider 接口，支持 DeepSeek / OpenAI / Claude 灵活切换和 fallback
2. **启用站点搜索**：集成 Hugo PaperMod 的搜索功能，支持按关键词/日期/标签检索历史日报
3. **源健康监控**：在 GitHub Actions 中增加源成功率统计，自动标记连续失败源
4. **摘要质量评分**：基于原文标题和生成摘要的一致性检测，自动标记低质量摘要

**中期（3-6 个月）**：
5. **全文分析**：对 select 阶段精选的 top 10 条目进行原文全文抓取，提升摘要质量
6. **通知推送**：支持 Email / Webhook 推送每日报告摘要
7. **周报/月报**：基于每日摘要积累生成更长周期的趋势回顾
8. **英文输出**：支持英文日报生成，覆盖更多国际源
9. **技术指标 Agent**：MACD/RSI/布林带等常用技术指标的计算和解读

**长期（6-12 个月）**：
10. **个性化配置**：Web UI 配置页面，用户可自定义关注源、关键词、报告频率
11. **多用户支持**：用户系统 + 个性化日报
12. **回测系统**：为 Agent 分析添加历史回测能力
13. **多模态输入**：支持图表截图 OCR（如财报电话会 PPT 截图自动提取关键数据）
14. **移动 App**：React Native 或 PWA，支持推送通知和离线阅读
